## 结构目录
`gsplat` 是一个高度优化的 3D Gaussian Splatting (3DGS) 库，它的核心设计理念是“底层用 CUDA 榨干显卡性能，上层用 Python 提供优雅的接口”。

### 1. `examples/` 

对于初学者，**不要一上来就看底层源码，先看例子**。这个目录包含了所有开箱即用的脚本。

* **`simple_trainer.py`**: 最核心的文件！这是一个完整且极简的 3DGS 训练脚本。如果你想用你的校园无人机数据训练模型，阅读这个文件可以了解如何加载数据、初始化高斯球、前向渲染、计算损失(Loss)以及反向传播更新高斯球。
* **`simple_viewer.py`**: 交互式可视化工具。用于在训练过程中或训练结束后，实时查看 3DGS 的渲染效果。
* **`image_fitting.py`**: 一个极其经典的“玩具案例”。它不处理 3D 场景，而是用一堆 2D 的高斯椭圆去拟合一张 2D 图片。**强烈建议你第一个看这个脚本**，它能帮你快速理解高斯分布是如何被优化器“推”到正确位置的，排除了 3D 相机投影的复杂性。
* **`datasets/`**: 包含如何读取各种数据集格式的脚本（比如 `colmap.py` 专门用来解析你通过 COLMAP 算出的稀疏点云和相机位姿）。

### 2. `gsplat/` (核心 Python 库 🧠)

这就是你 `import gsplat` 时实际调用的代码库。它扮演着 Python 和底层 CUDA 之间的桥梁。

* **`rendering.py`**: **极其重要**。这是前向渲染的核心入口。当你调用渲染函数生成图像时，代码就是从这里流向底层的。
* **`strategy/`**: 包含高斯球的**自适应控制策略**（即什么时候克隆、什么时候分裂、什么时候删除透明度低的高斯球）。
* `default.py`: 官方原始 3DGS 论文的启发式分裂/剪枝策略。
* `mcmc.py`: 一种基于马尔可夫链蒙特卡洛（MCMC）的更先进的生长策略。


* **`cuda/`**: **全村的希望，性能的来源**。
* `_backend.py` / `_wrapper.py`: Python 调用 CUDA 函数的接口文件（用 PyTorch 的 `autograd.Function` 封装，定义了前向传播和反向传播的梯度如何流动）。
* `csrc/` **(C++ / CUDA 源码)**: **这就是你之前编译失败卡住的地方！** 里面全是 `.cu` 和 `.cpp` 文件。包含了 3DGS 最硬核的数学和并行计算实现，比如：
* `Projection.cu`: 把 3D 的高斯球投影到 2D 屏幕上的核函数。
* `Rasterization.cu`: 也就是光栅化器，计算每个像素颜色值的超高速并行代码。
* `SphericalHarmonics.cu`: 计算球谐函数（SH），用来表示从不同视角看高斯球时颜色的变化。




* **`optimizers/`**: 针对 3DGS 定制的优化器（比如稀疏状态下的 Adam 优化器）。

### 3. `libs/` (新版架构的独立模块 🧩)

`gsplat` 团队在较新的版本中正在对代码进行重构和模块化，把一些通用功能抽离成了独立的子库。

* **`geometry/`**: 处理核心几何变换。比如 `pose.py` 和 `quaternion.py`，专门负责处理相机的位姿矩阵和四元数旋转的底层计算。
* **`scene/`**: 负责场景的管理，比如把成百上千万个高斯球的数据（位置、缩放、旋转、不透明度、球谐系数）打包在显存中高效管理。
* **`sensors/`**: 预留给相机模型或未来可能引入的激光雷达(LiDAR)等传感器模型的模块。

### 4. `tests/` (学习 API 用法的隐藏宝库 🛠️)

这里面全是使用 `pytest` 编写的单元测试文件（如 `test_math.py`, `test_rasterization.py`）。

* **作用**：如果你在看某个函数（比如球谐函数计算）不知道该传什么维度的 Tensor 进去，直接去 `tests/` 里找对应的测试文件。测试代码里展示的输入输出维度，是最完美的 API 文档。

### 5. `profiling/` (性能测试基准 ⏱️)

* 当你以后修改了底层的 CUDA 代码（比如你自己发明了一个更快的排序算法或光栅化算法），你需要用这里的脚本（如 `main.py`, `batch.py`）来测算你的代码比官方快了多少毫秒、显存占用了多少。

### 6. 根目录工程文件 ⚙️

* **`setup.py` / `pyproject.toml**`: Python 包的构建配置文件。里面详细定义了需要用 `nvcc` 编译哪些 `.cu` 文件。你之前运行 `pip install git+https...` 时，pip 就是在读取这两个文件来组织编译。
* **`docs/`**: 用于生成官方文档网站的 Markdown 和 reStructuredText 源码文件。

## 数据流

### 第一步：读取（Read） —— `Dataset` 是仓库保管员

`Dataset` 的职责非常单一，它不管模型训练，只管**找东西**。
在 PyTorch 中，一个合格的 `Dataset` 必须实现两个“祖传方法”：

1. `__len__(self)`：回答仓库里总共有多少件货物。在你的例子里，它会返回 **757**。
2. `__getitem__(self, idx)`：根据订单号（索引 `idx`）去货架上拿货。当你传入 `idx=5` 时，它会去硬盘里读取第 6 张图像，把它转成 Tensor，并且附带上这张图像的相机位姿（Pose）、内参（K）等标签，打包成一个字典返回。

> **特点**：`Dataset` 每次只拿**1个**样本。它绝对不会一次性把 757 张图全塞进内存（那样你的内存早爆了），它是**按需读取**的。

### 第二步：包装（Package） —— `DataLoader` 是物流车队

有了保管员还不够，GPU（训练循环）的胃口很大，一次只要1个样本太慢了，而且我们需要打乱顺序以防模型死记硬背。这时候就需要 `DataLoader` 出马了。

假设我们在 `simple_trainer.py` 里设置了 `batch_size=4`，`shuffle=True`，`num_workers=4`。`DataLoader` 会做以下事情：

1. **生成随机订单**：因为 `shuffle=True`，它会生成一个打乱的索引列表，比如 `[24, 756, 12, 489, ...]`。
2. **多线程搬运**：启动 4 个后台进程（`num_workers=4`），分别拿着前 4 个订单号找 `Dataset` 要数据。
3. **打包成批（Collate & Package）**：拿到这 4 张独立的图像 Tensor（每张形状是 `[3, H, W]`）后，`DataLoader` 会把它们“叠”在一起，拼接成一个形状为 **`[4, 3, H, W]` 的大 Tensor**（这就是一个 Batch）。
4. **送上高速**：如果设置了 `pin_memory=True`，它会把这个大 Tensor 放到特殊的锁页内存里，准备以极快的速度传输给 GPU。

### 第三步：分送（Deliver） —— 为什么用 `iter()` 和 `next()`？

这就回到了你最疑惑的地方。正常我们在做图像分类任务时，代码通常是这么写的：

```python
# 传统的 Epoch 驱动写法
for epoch in range(100):
    for data in dataloader:
        # 训练过程...

```

在上面这种写法中，`for` 循环其实在底层隐式地调用了 `iter()` 和 `next()`。当 757 张图被按 `batch_size=4` 取完（大约取 190 次）后，`dataloader` 就会抛出一个 `StopIteration` 信号，内层循环结束，表示“**1 个 Epoch（一轮）跑完了**”。

**但是！在 3DGS 或很多复杂的渲染任务中，训练往往是“Step 驱动”的。**
我们不在乎跑了几个 Epoch，我们只关心“我要整整迭代 30,000 步（Steps）”。所以，`simple_trainer.py` 采用了这种极其优雅的**手动取货模式**：

```python
trainloader_iter = iter(trainloader) # 1. 把物流车队变成一个“传送带”

for step in pbar: # 2. 明确我要跑 30000 步
    try:
        data = next(trainloader_iter) # 3. 按一下按钮，传送带送过来 1 个 Batch
    except StopIteration:
        # 4. 如果 757 张图在这步刚好用完了（传送带空了）
        trainloader_iter = iter(trainloader) # 重新启动传送带（下一轮打乱）
        data = next(trainloader_iter) # 继续拿货，完美衔接！

```

**使用 `iter` 的核心优势**：
它把 `DataLoader` 变成了一个**随叫随到、永不枯竭的“无限数据池”**。你的主训练循环（`for step in range(30000)`）永远不会被打断。不需要为了照顾 Epoch 的边界而去写复杂的嵌套循环，代码的控制权完全掌握在 `step` 手里。


> *附注：在 `gsplat` 的 `simple_trainer.py` 中，由于相机内参的尺寸差异或特殊处理，你会发现它通常强制设定 `batch_size = 1`。但哪怕是 1，`DataLoader` 依然在背后默默为你做着预读取和从 CPU 到 GPU 的锁页内存搬运工作。*

## gsplat 中的 exposure 选项与 depth 损失
### 1. `exposure`（曝光补偿参数）是干什么的？

在理想的学术数据集中（比如室内的合成数据集），相机的曝光是恒定不变的。但在真实的室外场景中（比如你的校园无人机航拍，或者自动驾驶采集车），由于面对太阳、进入树荫或隧道，相机的自动曝光（Auto-Exposure）会不断调整快门速度和 ISO，导致同一面墙在不同帧里看起来忽明忽暗。

* **如果不处理曝光**：3DGS 会“傻傻地”以为场景真的在发光或变暗，它会试图用球谐函数（SH）去拟合这些因为相机参数造成的明暗变化，最终导致场景中出现奇怪的颜色块（Floaters）或者多视角不一致的伪影。
* **代码的解决思路**：`data["exposure"]` 会读取照片的 EXIF 元数据（快门、ISO 等计算出的曝光值）。在后面的前向渲染阶段（`self.stage.render(..., exposure=exposure)`），渲染器会先学习一个“曝光无关（统一光照）”的纯净 3D 场景，然后利用这个 `exposure` 参数，人为地把渲染出来的纯净图像变暗或变亮，再去和真实采集到的（忽明忽暗的）`pixels` 计算 Loss。

### 2. 为什么引入了深度损失（Depth Loss）？

* **原始 3DGS 的致命弱点：几何稀烂**
如果只用 2D 图像的 RGB 差异去反推 3D，这就如同“看皮影戏猜真人的骨架”。对于有丰富纹理的地方还好，但遇到**纯色墙壁、无云的天空、柏油马路**等缺乏纹理的区域，3DGS 极其容易产生“Floaters（悬浮物）”——它会把一团高斯球放在镜头前很近的地方，只要从这个视角看颜色是对的就行，它不管深度对不对。这导致原始 3DGS 的深度图（Depth Map）往往是惨不忍睹的。
* **代码的解决思路：利用 COLMAP 的“免费”深度**
在训练 3DGS 前，我们必定会用 COLMAP（SfM 算法）算出初始的稀疏点云和相机位姿。COLMAP 在算位姿的同时，其实已经三角化出了一批**绝对准确的 3D 稀疏特征点**。
* `points`: 代表当前这幅 2D 图像中，那些能跟 COLMAP 3D 稀疏点对应上的像素坐标（比如 $M$ 个特征点）。
* `depths_gt`: COLMAP 算出的这 $M$ 个特征点在当前视角下的真实物理深度（Ground Truth Depth）。

* **它如何工作？**
如果开启了 `cfg.depth_loss`，渲染器不仅会渲染 RGB，还会顺便渲染出一张**深度图 (Depth Map)**。然后代码会根据 `points` 的坐标，把渲染出的深度值抽出来，与 `depths_gt` 算一个 L1 损失。相当于用一根无形的线，把那些乱飘的高斯球强行“拉”回到 COLMAP 算出的正确三维骨架上。
